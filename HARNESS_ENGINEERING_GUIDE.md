# Harness Engineering Guide

## 목적

이 문서는 OpenAI가 2026년 2월 11일 공개한 "하네스 엔지니어링: 에이전트 우선 세계에서 Codex 활용하기"와 2026년 2월 4일 공개한 "Codex 하네스 활용하기: OpenAI가 App Server를 구축한 방법"을 바탕으로, OpenAI가 하네스 엔지니어링을 실제로 어떻게 설계하고 사용하는지 정리한 실무 가이드다.

이 문서는 두 층으로 읽으면 된다.

- 1장과 2장은 OpenAI 공개 글의 사실 요약
- 3장 이후는 그 사실을 바탕으로 Codex 작업 환경에 적용하기 위한 실무 해석

핵심은 단순하다. 하네스 엔지니어링은 프롬프트를 예쁘게 다듬는 일이 아니라, 에이전트가 안정적으로 일할 수밖에 없게 만드는 실행 환경을 설계하는 일이다.

---

## 1. OpenAI가 말하는 하네스 엔지니어링

OpenAI 사례에서 하네스 엔지니어링은 다음을 뜻한다.

- 에이전트가 일하는 실행 환경을 설계한다
- 에이전트가 읽을 수 있는 형태로 지식을 리포지터리에 넣는다
- 아키텍처 경계와 품질 기준을 기계적으로 강제한다
- 검증, 리뷰, 복구, 정리 루프를 시스템 안에 넣는다
- 사람의 역할을 "직접 코딩"에서 "환경 설계와 의사결정"으로 이동시킨다

OpenAI는 이를 말로만 설명하지 않았다. 공개 글에 따르면 2025년 8월 말 빈 Git 리포지터리에서 시작해 약 5개월 동안 사람 손으로 코드 라인을 직접 쓰지 않고 내부 베타 제품을 만들었고, 약 백만 줄 규모의 코드, 약 1,500개의 PR, 내부 사용자와 외부 알파 테스터까지 운영했다.

즉 OpenAI의 하네스 엔지니어링은 "에이전트가 코드를 잘 쓰게 돕는 주변 기술"이 아니라, 소프트웨어 개발 시스템 자체를 에이전트 중심으로 다시 짠 것이다.

---

## 2. OpenAI는 실제로 무엇을 설계했는가

### 2.1 사람의 역할을 바꿨다

OpenAI 사례에서 사람은 더 이상 주된 구현자가 아니었다. 사람의 주요 업무는 아래로 이동했다.

- 목표를 더 작은 작업 단위로 분해하기
- 에이전트가 읽고 실행할 수 있는 형태로 문맥을 정리하기
- 빠진 도구, 추상화, 스크립트, 문서를 찾아 채워 넣기
- 결과를 검증하고 필요할 때만 판단을 제공하기

정리하면 "사람이 조정하고, 에이전트가 수행"하는 구조다.

### 2.2 빈 리포지터리부터 에이전트 친화적으로 시작했다

초기 스캐폴드는 Codex CLI와 GPT-5로 생성했다.

- 리포지터리 구조
- CI 설정
- 포맷팅 규칙
- 패키지 관리자 설정
- 애플리케이션 프레임워크
- 초기 `AGENTS.md`

중요한 점은 기존 사람이 쓴 레거시 코드 위에 에이전트를 얹은 것이 아니라, 처음부터 리포지터리를 에이전트가 다루기 쉬운 형태로 형성했다는 점이다.

### 2.3 실행 환경을 워크트리 단위로 격리했다

OpenAI는 Codex가 각 변경마다 독립적으로 앱을 실행하고 검증할 수 있게 했다.

- Git worktree마다 앱을 부팅
- 에이전트가 각 작업용 인스턴스를 직접 제어
- 작업이 끝나면 상태와 관측 데이터 정리

이 구조 덕분에 한 에이전트 실행이 다른 실행의 상태를 더럽히지 않는다. 하네스의 핵심은 모델보다 먼저 실행 환경의 충돌 가능성을 줄이는 데 있다.

### 2.4 UI를 에이전트가 읽을 수 있게 만들었다

OpenAI는 브라우저 자동화와 런타임 관찰을 에이전트의 기본 능력으로 넣었다.

- Chrome DevTools Protocol 연결
- DOM 스냅샷
- 스크린샷
- 탐색 동작을 위한 스킬

즉 에이전트가 "코드만 수정"하는 것이 아니라 아래를 직접 수행하게 만들었다.

- 버그 재현
- 수정 후 UI 검증
- 동작 변화 추론

### 2.5 관측 가능성을 에이전트에게 열어줬다

OpenAI는 로그, 메트릭, 추적을 로컬 관측 스택으로 노출했다.

- 로그 쿼리: LogQL
- 메트릭 쿼리: PromQL
- 워크트리별 격리된 관측 데이터

이렇게 되면 프롬프트가 "느리다" 수준에 머물지 않는다. 대신 아래처럼 검증 가능한 작업이 된다.

- 서비스 시작 시간을 800ms 이내로 줄이기
- 핵심 유저 여정에서 특정 스팬이 2초를 넘지 않게 만들기

즉 하네스는 에이전트가 추상적인 요구를 구체적인 운영 지표로 바꾸는 장치다.

### 2.6 `AGENTS.md`를 백과사전이 아니라 목차로 썼다

OpenAI가 초기에 배운 핵심 교훈은 이것이었다.

- 큰 단일 `AGENTS.md`는 실패한다

이유는 명확하다.

- 컨텍스트는 희소 자원이다
- 중요한 규칙과 덜 중요한 규칙이 섞인다
- 오래된 규칙이 쌓여 진실의 원천이 되지 못한다
- 커버리지, 신선도, 교차 링크를 기계적으로 검증하기 어렵다

그래서 OpenAI는 `AGENTS.md`를 짧은 맵으로 바꾸고, 실제 지식은 구조화된 `docs/`에 넣었다.

### 2.7 리포지터리 자체를 기록 시스템으로 만들었다

OpenAI 사례의 핵심은 "Codex가 볼 수 없는 것은 존재하지 않는 것과 같다"는 전제다.

에이전트가 접근할 수 없는 지식은 아래와 같이 사실상 무효다.

- Google Docs
- Slack 스레드
- 사람 머릿속의 암묵지

그래서 제품 원칙, 설계 근거, 기술 부채, 실행 계획 같은 정보를 리포지터리 안의 버전 관리되는 아티팩트로 이동시켰다.

공개 글에 나온 지식 저장소 예시는 대략 아래 구조다.

```text
AGENTS.md
ARCHITECTURE.md
docs/
  design-docs/
  exec-plans/
  generated/
  product-specs/
  references/
DESIGN.md
FRONTEND.md
PLANS.md
PRODUCT_SENSE.md
QUALITY_SCORE.md
RELIABILITY.md
SECURITY.md
```

여기서 중요한 점은 문서를 많이 만드는 것이 아니라 역할을 분리하는 것이다.

- `AGENTS.md`: 현재 컨텍스트에 넣을 짧은 맵
- `ARCHITECTURE.md`: 상위 구조와 레이어링
- `docs/exec-plans/`: 진행 중, 완료된 계획과 의사결정 로그
- `docs/references/`: LLM이 읽기 쉬운 외부 레퍼런스
- `QUALITY_SCORE.md`, `RELIABILITY.md`, `SECURITY.md`: 품질 상태와 목표

### 2.8 계획을 일회성 메모가 아니라 일급 아티팩트로 취급했다

OpenAI는 복잡한 작업을 진행할 때 실행 계획을 리포지터리에 저장했다.

- 작은 변경은 짧은 임시 계획
- 큰 변경은 버전 관리되는 실행 계획
- 진행 상황과 의사결정 로그를 함께 보관
- 알려진 기술 부채도 같은 체계 안에 보관

이렇게 해야 에이전트가 다음 실행에서 외부 설명 없이도 맥락을 이어갈 수 있다.

### 2.9 문서만 두지 않고 기계적으로 강제했다

OpenAI는 문서화가 중요하다고 말했지만, 문서만으로 충분하다고 보지 않았다.

- 전용 린터
- 구조적 테스트
- CI 작업
- 문서 신선도와 링크 상태 검증

특히 중요한 점은 린터 메시지 자체를 에이전트 친화적으로 설계했다는 것이다. 단순히 실패만 반환하는 것이 아니라, 에이전트가 다음 수정 행동을 알 수 있도록 오류 메시지에 수정 지침을 담았다.

즉 하네스는 규칙 파일만이 아니라 "실패 시 복구 경로까지 알려 주는 도구 체계"다.

### 2.10 아키텍처를 느슨하게 두지 않았다

OpenAI는 엄격한 레이어 구조를 채택했다. 공개 글의 예시는 아래와 같다.

- Types -> Config -> Repo -> Service -> Runtime -> UI
- 교차 관심사는 `Providers`라는 명시적 인터페이스로만 진입

핵심은 두 가지다.

- 허용된 의존성 방향을 아주 명확하게 만든다
- 나머지는 전부 금지하고 기계적으로 막는다

OpenAI의 주장은 분명하다. 사람 중심 조직에서는 이런 규칙이 과도하게 느껴질 수 있지만, 에이전트 처리량이 높아질수록 제약은 성능 전제 조건이 된다.

### 2.11 에이전트 가독성을 기준으로 기술 선택을 했다

OpenAI는 "에이전트가 내부화하고 추론할 수 있는가"를 기술 선택 기준으로 삼았다.

- 불투명한 외부 동작보다 리포지터리 안에서 완전히 이해 가능한 추상화 선호
- 종종 "지루한" 기술이 더 좋음
- 경우에 따라 공개 패키지 일부를 그대로 쓰는 것보다 필요한 부분을 재구현

기사의 예시로는 범용 동시성 제한 패키지를 가져오기보다, 계측과 테스트 요구에 맞는 자체 helper를 구현한 사례가 나온다.

즉 하네스 엔지니어링은 "최신 라이브러리 수집"보다 "에이전트가 디버그 가능한 구조 만들기"에 더 가깝다.

### 2.12 병합 철학도 처리량에 맞게 바꿨다

OpenAI는 PR 처리량이 높아지자 기존의 무거운 게이트가 병목이 된다고 봤다.

- PR은 짧게 유지
- 최소한의 차단 병합 게이트
- flaky test는 영구 차단보다 후속 실행으로 해결

다만 OpenAI도 이 모델이 모든 팀에 맞는다고 말하지는 않았다. 처리량이 낮은 조직, 사람이 주로 코딩하는 조직, 강한 규제 환경에서는 더 보수적인 게이트가 여전히 맞을 수 있다.

### 2.13 "에이전트 생성" 범위를 코드에만 두지 않았다

OpenAI는 에이전트가 아래를 모두 생성한다고 설명한다.

- 제품 코드와 테스트
- CI와 릴리즈 툴링
- 내부 개발자 도구
- 문서와 설계 내역
- 평가 하네스
- 리뷰 코멘트와 응답
- 리포지터리 유지 스크립트
- 프로덕션 대시보드 정의

즉 하네스 엔지니어링은 에이전트가 코드 파일만 건드리게 하는 체계가 아니라, 엔지니어링 시스템 전체를 다루게 하는 체계다.

### 2.14 자율성은 검증 루프를 인코딩할수록 올라갔다

OpenAI는 에이전트가 한 번의 프롬프트로 아래를 수행할 수 있다고 설명한다.

- 현재 상태 검증
- 버그 재현
- 실패 영상 녹화
- 수정 구현
- 앱 실행과 수정 검증
- 수정 후 영상 녹화
- PR 생성
- 피드백 대응
- 빌드 실패 수정
- 필요 시에만 사람에게 에스컬레이션
- 병합

여기서 핵심은 모델 자체가 갑자기 똑똑해진 것이 아니라, 개발 루프가 하네스 안으로 들어왔다는 점이다.

### 2.15 엔트로피를 백그라운드 작업으로 청소했다

완전 자율성에는 드리프트가 따른다. OpenAI도 초기에 금요일마다 수동으로 "AI 슬로프"를 치웠지만 확장되지 않았다고 말한다.

대신 아래를 리포지터리에 인코딩했다.

- 황금 원칙
- 정기적인 편차 검사
- 품질 등급 갱신
- 대상 리팩터링 PR 생성

즉 기술 부채 상환도 사람의 일회성 결심이 아니라 시스템 동작으로 바꿨다.

## 3. OpenAI 하네스에서 특히 중요한 설계 포인트

### 3.1 `AGENTS.md`는 짧고, 진짜 지식은 `docs/`에 둔다

이것이 OpenAI 사례에서 가장 실전적인 포인트다.

- `AGENTS.md`는 100줄 안팎의 맵
- 실제 지식은 분류된 문서
- 문서에는 소유자, 링크, 상태, 최신성이 있어야 함
- 문서 드리프트는 린터와 백그라운드 에이전트가 관리

즉 `AGENTS.md`는 하네스의 전부가 아니라, 하네스를 찾아가는 입구다.

### 3.2 지식은 "접근 가능성" 기준으로 본다

사람이 알고 있어도 에이전트가 접근할 수 없으면 하네스 바깥 지식이다.

따라서 아래는 하네스 관점에서 리스크다.

- 회의에서만 합의된 규칙
- Slack에만 있는 설계 결정
- 사람이 "원래 그렇게 하는 것"이라고 알고 있는 관행

반대로 아래는 하네스 자산이다.

- 버전 관리되는 Markdown
- 읽기 쉬운 schema와 examples
- 실행 계획
- 로컬에서 재현 가능한 스크립트

### 3.3 검증 수단이 있어야 자율성이 올라간다

OpenAI 사례에서 자율성의 원천은 더 긴 프롬프트가 아니었다.

- 브라우저 제어
- 로그/메트릭 쿼리
- 로컬 실행
- 구조적 테스트
- 리뷰 루프
- 승인 모델

즉 에이전트가 실제 세계를 관찰하고, 가설을 검증하고, 실패를 복구할 수 있을 때 자율성이 올라간다.

### 3.4 제약은 에이전트의 속도를 늦추는 것이 아니라 유지한다

OpenAI는 레이어, naming, logging, file size, 플랫폼 요구사항 같은 규칙을 맞춤형 린터로 강제했다.

사람 중심 워크플로에서는 귀찮아 보일 수 있지만, 에이전트 중심 워크플로에서는 아래 효과가 있다.

- 아키텍처 드리프트 억제
- 오류 메시지를 통해 수정 방향 즉시 제공
- 여러 실행에 동일 기준 적용
- 리뷰 비용 감소

### 3.5 하네스는 운영 중에 계속 진화해야 한다

OpenAI는 사람의 리뷰 코멘트와 버그를 문서 업데이트나 툴링 규칙으로 승격했다.

이 뜻은 명확하다.

- 반복되는 리뷰 지적은 문서가 아니라 규칙 후보
- 자주 발생하는 실수는 guardrail 후보
- 늘 같은 수정이 필요한 패턴은 백그라운드 정리 작업 후보

즉 하네스는 한 번 세팅하고 끝나는 설정 파일이 아니다.

---

## 4. OpenAI는 하네스를 제품 서피스에도 노출했다

OpenAI는 "하네스"를 리포지터리 운영 체계로만 쓰지 않았다. 같은 핵심 에이전트 루프를 여러 제품 서피스에서 재사용하기 위해 App Server라는 런타임 계층도 만들었다.

### 4.1 Codex 하네스 내부에 포함된 것

OpenAI가 설명한 Codex 하네스 내부 요소는 아래와 같다.

- 핵심 에이전트 루프
- 스레드 lifecycle과 persistence
- 설정과 인증
- 도구 실행
- MCP와 스킬 같은 확장

즉 하네스는 모델 호출 함수 하나가 아니라, 장기 실행되는 에이전트 런타임이다.

### 4.2 App Server로 같은 하네스를 여러 클라이언트에 연결했다

OpenAI는 같은 Codex 하네스를 아래 서피스에 재사용했다.

- CLI / TUI
- 웹 런타임
- VS Code 확장
- 데스크톱 앱
- 외부 파트너 통합

여기서 중요한 설계는 다음과 같다.

- 장기 실행 프로세스
- JSON-RPC 기반 양방향 프로토콜
- 스레드, 턴, 아이템이라는 명시적 프리미티브
- 승인이 필요하면 서버가 턴을 멈추고 클라이언트 응답 대기
- 스트리밍 이벤트를 UI 친화적인 이벤트로 변환

이 설계는 "에이전트 루프를 한 제품 안에 하드코딩하지 말고, 독립 실행 가능한 하네스로 분리하라"는 메시지로 읽을 수 있다.

### 4.3 우리에게 주는 시사점

프로젝트 하네스와 제품 하네스를 구분해야 한다.

- 프로젝트 하네스: repo 안에서 에이전트가 일하는 방식
- 제품 하네스: 같은 에이전트 루프를 여러 UI나 클라이언트에서 재사용하는 방식

초기에는 프로젝트 하네스만 잘 만들어도 충분하지만, 에이전트를 여러 서피스에 붙일 계획이라면 OpenAI처럼 런타임과 프로토콜을 분리하는 설계가 필요하다.

---

## 5. 이 문서를 Codex 작업 환경에 번역하면

OpenAI 사례를 그대로 복사할 필요는 없지만, 아래 순서는 거의 그대로 가져올 수 있다.

### 5.1 최소 1단계: 얇은 `AGENTS.md`

루트 `AGENTS.md`에는 아래 정도만 둔다.

- 프로젝트 목표
- 수정 원칙
- 금지 영역
- 검증 명령
- 문서 지도를 가리키는 링크

### 5.2 최소 2단계: `docs/`를 진실의 원천으로 승격

최소 구조는 이 정도면 충분하다.

```text
docs/
  architecture/
  product/
  exec-plans/
  references/
  reliability/
  security/
```

중요한 것은 문서 수가 아니라 다음이다.

- 어디에 무엇이 있는지 예측 가능해야 함
- 에이전트가 검색 가능한 이름이어야 함
- 오래된 문서를 잡아내는 규칙이 있어야 함

### 5.3 최소 3단계: 실행을 격리

가능하면 아래를 갖춘다.

- 작업 단위별 worktree 또는 독립 실행 환경
- 독립 `.env` 또는 설정 주입 방식
- 작업 종료 시 정리 가능한 상태

이 격리가 없으면 에이전트는 재현성과 디버깅에서 금방 흔들린다.

### 5.4 최소 4단계: 에이전트가 관찰할 수 있게 한다

최소한 아래 중 일부는 직접 열어줘야 한다.

- 브라우저 자동화
- 애플리케이션 로그
- 테스트 결과
- 빌드 상태
- 메트릭 또는 tracing

에이전트가 직접 볼 수 없는 문제는 결국 사람 설명에 의존하게 되고, 그러면 하네스가 약해진다.

### 5.5 최소 5단계: 맞춤형 실패 메시지를 가진 강제 규칙

가능하면 아래 순서로 도입한다.

1. formatter
2. lint
3. typecheck
4. 구조적 테스트
5. 경계 위반 검사

이때 오류 메시지는 사람이 읽기 좋게만 쓰지 말고, 에이전트가 다음 액션을 알 수 있게 써야 한다.

### 5.6 최소 6단계: 리뷰 지적을 규칙으로 승격

같은 지적이 두세 번 반복되면 아래 중 하나로 승격한다.

- `AGENTS.md`
- `docs/`
- 스킬
- 린터
- CI 검사
- 백그라운드 정리 작업

## 6. OpenAI 사례를 기준으로 본 권장 하네스 구조

### 전역 레이어

- 기본 모델, reasoning, 승인 정책
- 공통 MCP
- 공통 브라우저/문서/리뷰 스킬

### 프로젝트 레이어

- `docs/` 지식 베이스
- 검증 명령
- 아키텍처 규칙
- 브랜치/PR/병합 정책

### 실행 레이어

- worktree 또는 task별 실행 환경
- 브라우저 및 앱 제어 도구
- 로그, 메트릭, 추적 노출
- 정리 스크립트

### 강제 레이어

- lint
- typecheck
- 구조적 테스트
- CI
- 권한/승인 정책

### 유지보수 레이어

- tech debt tracker
- 품질 점수표
- doc-gardening
- 정기 리팩터링 PR

## 7. 안티패턴

OpenAI 사례를 기준으로 보면 아래는 좋지 않다.

### 안티패턴 1. 거대한 `AGENTS.md` 하나로 끝내려 한다

문제:

- 컨텍스트 낭비
- 오래된 규칙 누적
- 기계적 검증 불가

대안:

- 짧은 `AGENTS.md`
- 구조화된 `docs/`

### 안티패턴 2. 사람만 아는 문맥을 방치한다

문제:

- 에이전트는 같은 실수를 반복
- 새 실행마다 설명 비용 발생

대안:

- 설계 결정과 규칙을 repo 안 Markdown으로 이동

### 안티패턴 3. 검증 루프 없이 코드만 생성한다

문제:

- 버그 재현과 수정 검증이 분리
- 사람 QA가 병목

대안:

- 브라우저
- 로그
- 메트릭
- 테스트
- 승인 루프

### 안티패턴 4. 문서화만 있고 강제 장치가 없다

문제:

- 아키텍처 드리프트
- 리뷰 비용 누적

대안:

- 맞춤형 lint
- 구조적 테스트
- CI

### 안티패턴 5. 정리 작업을 사람 의지에 맡긴다

문제:

- AI 슬로프 누적
- 금요일 대청소 문화 형성

대안:

- 품질 점수
- 정기 청소 PR
- 황금 원칙 자동 검사

## 8. 실무 체크리스트

새 프로젝트에 하네스를 붙일 때는 아래 순서로 점검한다.

### 기본 구조

- 루트 `AGENTS.md`가 짧은 맵 역할을 하는가
- `docs/`가 주제별로 구조화되어 있는가
- 실행 계획을 저장할 위치가 있는가

### 실행 환경

- 작업별 격리 실행이 가능한가
- 앱을 에이전트가 직접 실행하고 종료할 수 있는가
- 브라우저 검증이 가능한가

### 관찰 가능성

- 로그를 읽을 수 있는가
- 테스트 결과를 프로그램적으로 해석할 수 있는가
- 성능/신뢰성 지표를 쿼리할 수 있는가

### 강제 장치

- 경계 위반을 잡는 lint나 테스트가 있는가
- 검증 실패가 명확한 수정 지침을 주는가
- 문서 드리프트를 검출하는가

### 유지보수

- 기술 부채를 쌓아 두는 대신 추적하는가
- 반복 리뷰 지적을 규칙으로 승격하는가
- 정기적인 정리 작업이 있는가

---

## 9. 결론

OpenAI 사례에서 하네스 엔지니어링은 세 가지를 의미했다.

- 에이전트가 일할 실행 환경을 설계한다
- 에이전트가 읽을 수 있는 지식 체계를 리포지터리에 만든다
- 경계, 검증, 정리, 승인 루프를 시스템 안에 넣는다

따라서 하네스 엔지니어링의 핵심 질문은 "어떤 프롬프트가 좋을까?"가 아니다.

- 에이전트가 무엇을 볼 수 있는가
- 무엇을 직접 실행하고 검증할 수 있는가
- 어떤 규칙이 기계적으로 강제되는가
- 어떤 드리프트가 자동으로 청소되는가

이 기준으로 보면 좋은 하네스는 모델을 더 똑똑하게 만드는 것이 아니라, 에이전트가 실수하기 어렵고 다음 실행에서 더 나아지기 쉬운 시스템을 만든다.
